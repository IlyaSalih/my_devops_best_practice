# Управление процессами
---
## Инструменты быстрого обзора
```bash
top                          # интерактивный мониторинг (q — выйти)
htop                         # удобнее top, если установлен
ps aux                       # снимок всех процессов
ps aux --sort=-%cpu          # отсортировать по CPU
ps aux --sort=-%mem          # отсортировать по памяти
ps -ef                       # дерево процессов (PPID виден)
pstree -p                    # дерево с PID
```
### Что читать в выводе `ps aux`:
```
USER       PID  %CPU  %MEM    VSZ    RSS  STAT  COMMAND
www-data  1234  98.0   2.1  512000  85000  R    /usr/bin/php-fpm
```
`VSZ` — виртуальная память (включая shared libs, не показатель реального потребления)
`RSS` — реальная физическая память процесса
`STAT`: `R` running · `S` sleeping · `D` uninterruptible sleep (I/O wait) · `Z` zombie · `T` stopped

## Сценарий 1: Высокая нагрузка на CPU
### Симптом: сервер тормозит, load average высокий.
```bash
# Шаг 1 — найти виновника
ps aux --sort=-%cpu | head -10

# Шаг 2 — посмотреть что именно делает процесс
top -p <PID>                 # следить за конкретным PID
strace -p <PID> -c           # статистика системных вызовов (что делает)
strace -p <PID>              # живой поток syscall (Ctrl+C — остановить)

# Шаг 3 — понять сколько времени процесс уже жрёт CPU
ps -p <PID> -o pid,etime,%cpu,cmd
# etime — время жизни процесса, поможет понять: это пик или хроническая проблема

# Шаг 4 — проверить не застрял ли процесс в I/O
ps aux | grep " D "          # состояние D = ждёт I/O (диск, сеть, NFS)
iotop -p <PID>               # I/O активность процесса (если установлен)
```
### Решение:
```bash
# Снизить приоритет (не убивать, но дать дышать другим)
renice +10 -p <PID>          # повысить nice value = снизить приоритет CPU
renice +10 -u www-data       # для всех процессов пользователя

# Убить если необходимо (см. раздел про сигналы)
kill -15 <PID>               # сначала SIGTERM — дать шанс завершиться
kill -9 <PID>                # SIGKILL — если не реагирует
```

## Сценарий 2: Утечка памяти
### Симптом: свободная память постепенно заканчивается, после рестарта сервиса — временно освобождается.
```bash
# Шаг 1 — кто занимает память
ps aux --sort=-%mem | head -10
free -h                      # общая картина (buff/cache не тревожит — это норма)

# Шаг 2 — детально по процессу
cat /proc/<PID>/status | grep -i vm
# VmRSS — физическая память
# VmSwap — сколько ушло в swap (плохой знак)

# Шаг 3 — следить за динамикой
watch -n 2 'ps -p <PID> -o pid,rss,%mem,cmd'
# если RSS растёт со временем — утечка памяти

# Шаг 4 — проверить swap
swapon --show                # использование swap
vmstat 2 5                   # si/so: swap in / swap out (не должны быть постоянно > 0)
```
### Решение:
```bash
# Краткосрочно — рестарт сервиса
systemctl restart <service>

# Если памяти критически мало прямо сейчас
echo 3 > /proc/sys/vm/drop_caches   # очистить page cache (безопасно, временно)

# Долгосрочно — настроить ограничения в systemd unit
# /etc/systemd/system/<service>.service
# [Service]
# MemoryMax=512M               # убьёт процесс при превышении
# MemoryHigh=400M              # притормозит при приближении к лимиту
```

## Сценарий 3: Процесс завис и не отвечает
### Симптом: сервис не отвечает, но процесс есть в `ps aux`.
```bash
# Шаг 1 — состояние процесса
ps -p <PID> -o pid,stat,wchan,cmd
# wchan — в каком kernel call завис
# stat D = завис в I/O (нельзя убить через kill -9 пока не завершится I/O)
# stat S = спит (нормально, убивается)

# Шаг 2 — что держит процесс
strace -p <PID>              # какой syscall завис
lsof -p <PID>                # открытые файлы, сокеты, дескрипторы
lsof -p <PID> | grep -i "REG\|IPv"  # только файлы и сетевые соединения

# Шаг 3 — не ждёт ли он другой процесс или ресурс
cat /proc/<PID>/wchan        # имя kernel function где завис
ls -la /proc/<PID>/fd        # открытые файловые дескрипторы
```
### Решение:
```bash
kill -15 <PID>               # попробовать SIGTERM
sleep 5 && kill -9 <PID>     # если через 5 сек не умер — SIGKILL

# Если stat=D (завис в I/O) — kill -9 не поможет
# Нужно устранить причину I/O (недоступный NFS, зависший диск)
# Иногда единственный выход — перезагрузка
```

# Сигналы: SIGTERM vs SIGKILL vs SIGHUP
```
SIGTERM	(15) - Вежливый запрос завершиться. Процесс может поймать, закончить работу, почистить за собой. Используй первым.
SIGKILL	(9) - Принудительное уничтожение ядром. Процесс не может перехватить. Не оставляет времени на cleanup.
SIGHUP	(1) - Изначально — разрыв соединения. Большинство демонов перехватывают его как сигнал перечитать конфиг без рестарта.
SIGINT	(2) - То же что Ctrl+C в терминале.
SIGSTOP	(19) - Заморозить процесс (нельзя перехватить).
SIGCONT	(18) - Продолжить замороженный процесс.
```
```bash
kill -15 <PID>               # SIGTERM — всегда начинай с него
kill -1 <PID>                # SIGHUP — reload конфига (nginx, sshd, etc.)
kill -9 <PID>                # SIGKILL — только если SIGTERM не помог
kill -19 <PID>               # заморозить процесс
kill -18 <PID>               # разморозить

killall nginx                # SIGTERM всем процессам с именем nginx
killall -9 nginx             # SIGKILL всем
pkill -f "php-fpm"           # по части имени или аргументов
pkill -u www-data            # все процессы пользователя

# Правильная последовательность при ручном завершении:
kill -15 <PID> && sleep 5 && kill -9 <PID> 2>/dev/null
```
> `kill -9` не даёт процессу закрыть файлы, соединения с БД, дописать логи. После него возможны битые файлы и незакрытые транзакции.

## Zombie и Orphan процессы
## Zombie (defunct)
### Что это: процесс завершился, но родитель не забрал его exit code через `wait()`. Запись в таблице процессов остаётся.
```bash
# Найти zombie
ps aux | grep -w Z
ps aux | grep defunct

# Zombie не занимают CPU/память — только PID
# Убить zombie напрямую нельзя (он уже мёртв)

# Решение — найти и перезапустить родителя
ps -p <ZOMBIE_PID> -o ppid=     # узнать PID родителя
kill -15 <PPID>                  # завершить родителя — zombie исчезнут
# Если родитель — важный сервис, перезапусти его через systemctl

# Много zombie = баг в приложении (не обрабатывает SIGCHLD)
```
## Orphan (сирота)
### Что это: родительский процесс умер, дочерний продолжает работать. Ядро автоматически переназначает его под `init` (PID 1).
```bash
# Найти осиротевшие процессы (PPID = 1)
ps -eo pid,ppid,cmd | awk '$2 == 1'

# Orphan процессы обычно не проблема — init за ними присмотрит
# Проблема если orphan — это утечка (процесс не должен был остаться жить)
kill -15 <PID>
```

## Глубокая диагностика: strace и lsof
### strace — что делает процесс на уровне ядра
```bash
strace -p <PID>              # подключиться к живому процессу
strace -p <PID> -c           # статистика по syscall (что занимает время)
strace -p <PID> -e trace=network   # только сетевые вызовы
strace -p <PID> -e trace=file      # только файловые операции
strace -p <PID> -e trace=read,write # только чтение/запись
strace -p <PID> -T           # время каждого syscall
strace -p <PID> -o /tmp/out.txt    # сохранить в файл
strace -f -p <PID>           # следить за дочерними процессами тоже

# Запустить программу через strace
strace nginx -t 2>&1 | grep -i error
```
### Типичные паттерны в выводе strace:
```
# Процесс ждёт данных из сети:
epoll_wait(5, ..., 30000)    — застрял в ожидании событий

# Процесс в бесконечном цикле чтения:
read(3, ...) = 0             — читает EOF снова и снова (баг)

# Процесс не может открыть файл:
openat(...) = -1 ENOENT      — файл не найден
openat(...) = -1 EACCES      — нет прав
```
lsof — что открыто у процесса
```bash
lsof -p <PID>                # всё открытое (файлы, сокеты, pipes)
lsof -p <PID> | grep REG     # только обычные файлы
lsof -p <PID> | grep IPv     # только сетевые соединения
lsof -p <PID> | grep DEL     # удалённые файлы (процесс держит дескриптор)

# Найти процесс занявший порт
lsof -i :80
lsof -i :8080
lsof -i TCP:443

# Найти процесс держащий файл
lsof /var/log/app.log
lsof /var/lib/mysql/

# Файл удалён но место не освободилось — процесс держит дескриптор
lsof | grep deleted
# Решение: рестарт процесса или > /proc/<PID>/fd/<FD>
```

## Шпаргалка: диагностика за 2 минуты
```bash
# 1. Кто грузит систему?
ps aux --sort=-%cpu | head -5
ps aux --sort=-%mem | head -5

# 2. Есть ли zombie?
ps aux | grep -c defunct

# 3. Что делает подозрительный процесс?
strace -p <PID> -c -e trace=all   # 10 сек, потом Ctrl+C

# 4. Что он держит открытым?
lsof -p <PID> | grep -E "REG|IPv|DEL"

# 5. Завершить правильно
kill -15 <PID> && sleep 3 && kill -9 <PID> 2>/dev/null
```
